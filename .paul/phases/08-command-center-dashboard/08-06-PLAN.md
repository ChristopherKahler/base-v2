<objective>
## Goal
Bring the Usage Analytics panel to feature parity with TokenBBQ — expanded data model, project-level aggregation, daily breakdown table, and provider-abstracted backend ready for multi-platform support in Phase 9.

## Purpose
The current Usage Analytics shows summary cards, one chart, model list, and a session table. TokenBBQ shows a richer picture: projects table, daily breakdown with cache columns, cost/day, active days, monthly trends. Reaching parity makes the dashboard a credible standalone tool. More importantly, the backend currently hardcodes Claude Code JSONL parsing — Phase 9 will add Codex, Gemini, OpenCode, Cursor, etc. This plan restructures the parsing layer so each provider is a self-contained module normalizing to a shared `UsageEvent` type, making Phase 9 a drop-in exercise.

## Output
- Expanded Rust structs + new API endpoint (`/api/usage/projects`)
- Provider-abstracted parsing module (Claude Code as first provider)
- Projects table + Daily Breakdown table in frontend
- Enriched stat cards (Cost/Day, Active Days)
</objective>

    **Restructure parse_usage_data into a provider-based architecture:**

    1. Define a `UsageEvent` struct (the normalized unit all providers produce):
       ```
       struct UsageEvent {
           provider: String,      // "claude-code", "codex", "gemini", etc.
           model: String,
           input_tokens: u64,
           output_tokens: u64,
           cache_read: u64,
           cache_write: u64,
           cost: f64,
           timestamp: String,     // ISO 8601
           project: String,
           session_id: String,
       }
       ```

    2. Extract current Claude Code JSONL parsing into `fn parse_claude_code(days: u32) -> Vec<UsageEvent>`:
       - Same logic as current parse_usage_data's inner loop
       - Reads ~/.claude/projects/**/*.jsonl
       - Resolves project name from directory path (existing approach)
       - Uses existing estimate_cost() for pricing

    3. Create `fn collect_all_events(days: u32) -> Vec<UsageEvent>`:
       - Calls parse_claude_code(days)
       - Future: will also call parse_codex(), parse_gemini(), etc.
       - Returns merged + sorted Vec

    4. Refactor aggregation into `fn aggregate_usage(events: &[UsageEvent]) -> (UsageSummary, Vec<SessionUsageEntry>)`:
       - Same aggregation logic but reads from UsageEvent instead of raw JSONL
       - This function is provider-agnostic

    **Expand DailyUsage struct:**
    - Add `cache_read: u64`, `cache_write: u64`, `sources: usize` (event count for the day)
    - Update daily_map aggregation to track these fields

    **Expand UsageSummary struct:**
    - Add `active_days: u32` (count of unique dates in daily array)
    - Add `cost_per_day: f64` (estimated_cost_usd / active_days, or 0 if no active days)
    - Add `top_model: String` (model with highest token count)

    **Add ProjectUsage struct:**

    ```
        project: String,
        provider: String,
        total_tokens: u64,
        cost: f64,
        event_count: usize,
        last_active: String,
    }
    ```

    **Add new endpoint:**
    - `GET /api/usage/projects?days=N` → returns `Vec<ProjectUsage>` sorted by cost desc

    **Wire route in server.rs:**
    - `.route("/api/usage/projects", get(super::api::usage_projects))`

    Avoid: Breaking existing /api/usage/summary or /api/usage/sessions response shapes — add fields, don't remove or rename.
    Avoid: Actually implementing other providers — just design the abstraction. Claude Code is the only parser for now.
