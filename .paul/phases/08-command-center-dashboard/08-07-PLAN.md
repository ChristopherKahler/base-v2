## Goal
Complete task CRUD (edit, delete) + task detail panel + create task modal — the Operations tab backbone that every subsequent polish plan builds on.

## Purpose

This plan closes the CRUD gap and adds the detail panel that makes tasks inspectable and mutable — the fundamental interaction model for a project management surface.

- 2 new frontend API functions
- Task detail slide-out panel (click card → expand, inline edit, delete)
- Create task modal (replaces inline form bar, adds priority + description fields)

<acceptance_criteria>

## AC-1: Task Update Endpoint
```gherkin
Given a task exists in the graph with a known IRI
When PATCH /api/ops/task/{iri} is called with { name?, priority?, description? }
Then the task's fields are updated in the graph and persisted to disk
And the response returns the updated task object
```

## AC-2: Task Delete Endpoint
```gherkin
Given a task exists in the graph with a known IRI
When DELETE /api/ops/task/{iri} is called
Then all triples for that task are removed (type, name, status, priority, description, belongsTo, hasTask edges)
And the response confirms deletion
And the task no longer appears in GET /api/ops/projects
```

## AC-3: Task Detail Panel
```gherkin
Given the Operations panel is loaded with tasks visible
When a user clicks a kanban card or table row
Then a detail panel slides in from the right showing the task's full details
And the user can inline-edit the task name (click to edit, Enter to save)
And the user can change priority via a dropdown
And the user can change status via a dropdown (not just drag)
And the user can edit/add a description (textarea, auto-save on blur)
And a delete button with confirmation removes the task and closes the panel
```

## AC-4: Create Task Modal
```gherkin
Given the user clicks the "+ Task" button in the Operations header
When the create task modal opens
Then the modal contains fields for: name (required), project (required dropdown), priority (optional dropdown, default normal), description (optional textarea)
And submitting creates the task with all fields populated
And the modal closes and the kanban/table updates with the new task
```

  <name>Task 1: Backend — Task update and delete endpoints</name>
  <files>src/dashboard/api.rs, src/dashboard/server.rs</files>
  <action>
    **In api.rs:**

    1. Add `description` field to `OpsTask` struct (String, default empty)
    2. Update `ops_projects` SPARQL to query `OPTIONAL { ?t {p}:description ?tdesc }`
    3. Add `description` field to `CreateTaskBody` (Option<String>)
    4. Update `create_task` to insert description triple when provided

    5. Create `UpdateTaskBody` struct (not to be confused with `UpdateStatusBody`):
       ```
       pub struct UpdateTaskBody {
           pub name: Option<String>,
           pub priority: Option<String>,
           pub description: Option<String>,
       }
       ```

    6. Create `update_task` handler:
       - Route: PATCH /api/ops/task/{iri}
       - Accepts UpdateTaskBody
       - For each Some field: DELETE old triple + INSERT new triple (same pattern as update_task_status)
       - Validate priority values: ["low", "normal", "high", "urgent"]
       - Write back to disk
       - Return updated task fields

    7. Create `delete_task` handler:
       - Route: DELETE /api/ops/task/{iri}
       - Delete ALL triples where task IRI is subject: type, name, status, priority, description, belongsTo
       - Delete ALL triples where task IRI is object: hasTask edges from project
       - Use: DELETE { GRAPH ?g { <{iri}> ?p ?o } } WHERE { GRAPH ?g { <{iri}> ?p ?o } }
       - Then: DELETE { GRAPH ?g { ?s ?p <{iri}> } } WHERE { GRAPH ?g { ?s ?p <{iri}> } }
       - Write back to disk
       - Return { deleted: true, iri }

    **In server.rs:**
    - Add route: .route("/api/ops/task/{iri}", patch(super::api::update_task))
    - Add route: .route("/api/ops/task/{iri}", delete(super::api::delete_task))
    - Keep existing /api/ops/task/{iri}/status route (backwards compatible)

    **Avoid:**
    - Do NOT remove the existing update_task_status endpoint — it's used by drag-drop
    - Do NOT change the OpsProject or OpsMilestone structs
  </action>
  <verify>
    cargo build succeeds with no errors.
    Start dashboard, confirm /api/ops/projects returns tasks with description field (empty string default).
  </verify>
  <done>AC-1 and AC-2 satisfied: Task update and delete endpoints compile and respond correctly</done>
</task>

    ```
    ┌─────────────────────────────────────────────┐
    │ Task Detail                            [✕]  │
    ├─────────────────────────────────────────────┤
    │ Name: [editable — click to edit, blur save] │
    │ Project: {project name} (read-only label)   │
    │ Status: [dropdown: active/blocked/done/...] │
    │ Priority: [dropdown: low/normal/high/urgent]│
    │                                             │
    │ Description                                 │
    │ [textarea — auto-save on blur]              │
    │                                             │
    │                                             │
    │ ─────────────────────────────────────────── │
    │ [🗑 Delete Task]                            │
    └─────────────────────────────────────────────┘
    ```

    5. Editable name: click on name text → becomes input, blur/Enter → call updateTask, update local state
    6. Status dropdown: on:change → call updateTaskStatus (existing), update local state
    7. Priority dropdown: on:change → call updateTask({ priority }), update local state
    8. Description textarea: on:blur → call updateTask({ description }), update local state
    9. Delete button: on:click → confirm dialog (custom styled div, NOT window.confirm/alert) → call deleteTask → remove from local state → close panel
    10. Close button (✕): set selectedTask = null

    **Layout adjustment:**
    - When detail panel is open, the main kanban/table area shrinks (same pattern as GraphExplorer)
    - Kanban/table area gets `flex: 1` and detail panel gets fixed width (~320px)
    - Wrap main-content in a flex row container

    - `.task-detail-panel` — fixed width 320px, right side, border-left, background surface-01, overflow-y auto
    - `.task-detail-header` — flex between title and close button
    - `.task-detail-field` — label + value pairs, consistent spacing
    - `.task-detail-name` — editable text, click state shows input
    - `.task-detail-textarea` — description textarea, dark background, no resize, auto-height
    - `.task-detail-delete` — red text button, bottom of panel
    - `.task-detail-confirm` — confirmation overlay with Cancel/Delete buttons (NOT browser alert)
    - Transition: panel slides in from right (transform + opacity transition)

    **Avoid:**
    - Do NOT use window.alert() or window.confirm() — use custom styled confirmation UI
    - Do NOT break drag-and-drop — card click vs drag must coexist (click fires on mouseup without drag movement)
    - Do NOT re-fetch all projects on every field change — optimistic local updates, re-fetch only on failure
