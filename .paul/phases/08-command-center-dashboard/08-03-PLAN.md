<objective>
## Goal
Add WebSocket-powered session activity panel and drag-and-drop kanban status updates to the Command Center Dashboard.

## Purpose
Session Activity makes the hook engine visible — operators can see what BASE is doing in real-time (which hooks fire, which domains match, what gets injected vs suppressed). Drag-drop kanban turns the operations panel from read-only display into an interactive task management surface. Both transform the dashboard from observer to operator tool.

## Output
- `SessionActivity.svelte` — real-time event stream panel via WebSocket
- Hook event JSONL log (hooks append events, server tails + pushes)
- WebSocket upgrade handler in axum server
- PATCH endpoint for task status updates
- Drag-drop card movement on kanban columns
</objective>

## AC-1: WebSocket connection delivers hook events
```gherkin
Given the dashboard is running and Session Activity panel is open
When a Claude Code hook fires (session-start, user-prompt-submit, pre-tool-use, post-tool-use)
Then the event appears in the Session Activity feed within 1 second
```

## AC-2: Hook events persist to JSONL log
```gherkin
Given hooks are configured and a Claude Code session is active
When any hook fires while the dashboard server is NOT running
Then hook events are still appended to the JSONL log file
And when the dashboard server starts, it loads recent history from the log
```

## AC-3: Drag-drop updates task status in graph
```gherkin
Given the Operations panel is in kanban view
When a task card is dragged from one status column to another
Then a PATCH request updates the task status in the graph
And the card renders in its new column without a full page refresh
```

<task type="auto">
  <name>Task 1: Hook event JSONL logging + WebSocket server endpoint</name>
  <files>src/dashboard/server.rs, src/dashboard/api.rs, src/dashboard/mod.rs, src/hook/mod.rs</files>
  <action>
    **Hook event logging:**
    - In `src/hook/mod.rs`, after each hook handler completes, append a JSON line to `.base/hook-events.jsonl`
    - Each line: `{"ts": "ISO8601", "hook": "session-start|user-prompt-submit|pre-tool-use|post-tool-use", "domains_matched": [...], "rules_injected": N, "suppressed": N, "prompt_num": N}`
    - Append is fire-and-forget — if log write fails, hook still succeeds (fail-open)
    - Use `OpenOptions::append(true).create(true)` — no lock needed for single-writer append

    **WebSocket endpoint:**
    - Add `/api/ws/session` WebSocket upgrade route in `server.rs`
    - Use `axum::extract::ws::{WebSocket, WebSocketUpgrade}` (ws feature already enabled)
    - On connect: send last 100 lines of `hook-events.jsonl` as initial backfill
    - Then tail the file: use `tokio::time::interval(500ms)` to poll file length, push new lines
    - On disconnect: clean up watcher task
    - Add `ws_session` handler function in `api.rs` (or new `ws.rs` submodule if cleaner)

    **File tailing approach:**
    - Store last-read byte offset per connection
    - Every 500ms: check file size, if grown → read new bytes → split into lines → send as WS text frames
    - Simple and robust — no inotify/fswatch dependency

    Avoid: tokio::fs::File for the append path in hooks — hooks run in sync context. Use std::fs.
    Avoid: Broadcasting to all WS clients from hooks directly — decouple via file.
  </action>

    - `cargo test` passes
    - `cargo build --release` succeeds
    - Manual: start dashboard, trigger a hook (run `echo '{}' | base hook session-start`), verify event appears at ws://localhost:3741/api/ws/session via wscat or browser devtools
  </verify>

    - SPARQL UPDATE: delete old status triple, insert new one
    - Write back to trig file (same Mutex<Store> + write_back pattern as OperatorNotes)
    - Return 200 with updated task object on success, 404 if IRI not found

    **Drag-drop in OperationsPanel.svelte:**
    - Add HTML5 drag-and-drop to kanban cards:
      - `draggable="true"` on task cards
      - `on:dragstart` → store task IRI + source column
      - `on:dragover` → `event.preventDefault()` on column containers (allow drop)
      - `on:dragenter` → add visual drop indicator (highlight column border)
      - `on:dragleave` → remove drop indicator
      - `on:drop` → call PATCH endpoint with new status, update local state optimistically
    - Optimistic UI: move card immediately, revert if PATCH fails
    - Visual feedback: dragged card gets opacity 0.5, drop target column gets accent border glow
    - Map column IDs to status values: active → "active", in_progress → "in_progress", blocked → "blocked", completed → "completed"
