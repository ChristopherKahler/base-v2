<objective>
## Goal
Rules become graph entities with relational edges. Domain matching splits into prompt-keywords (natural language, user-configured) and file-keywords (code-oriented, system-suggestable). Rule injection queries the graph for matched domain's rules AND 1-hop neighborhood (related projects, decisions, entities) — surgical precision with relational comprehension.

## Purpose
This is the architectural foundation for CARL absorption. Moving rules from static TOML text to graph entities unlocks the relational power of the ontology — a domain match doesn't just dump rules, it surfaces the neighborhood. Every subsequent Phase 6 plan (DEVMODE, context brackets, base learn) depends on rules-in-graph.

## Output
- Updated `domains.toml` schema with `prompt_keywords` + `file_keywords`
- New `base domain sync` command that creates Domain/Rule entities in graph
- Graph-backed injection in `user_prompt_submit` (SPARQL replaces struct read)
- Decision entities migrated from `carl.json` into graph (edges to domains)
</objective>

<task type="auto">
  <name>Task 2: Domain/Rule/Decision graph sync</name>
  <files>src/domain/sync.rs, src/cli.rs, tests/domain_sync.rs</files>
  <action>
    Create new module src/domain/sync.rs:

    Function `sync_domains_to_graph(config: &BaseConfig, cwd: &Path, carl_json_path: Option<&Path>) -> Result<SyncStats>`:

    1. Load workspace graph (existing store.rs pattern)
    2. Read domains.toml via existing load_domains()
    3. For each DomainDef:
       - Build IRI: `{ns_uri}domain/{slug}` (e.g., `ops:domain/development`)
       - DELETE existing triples for this domain IRI (idempotent)
       - INSERT domain entity:
         - rdf:type ops:Domain
         - ops:name "{domain.name}"
         - ops:mode "{domain.mode}"
         - For each prompt_keyword: ops:triggerKeyword "{kw}"
         - For each file_keyword: ops:triggerKeyword "{kw}" (with a type marker, or separate predicate ops:fileKeyword)
         - For each path: ops:triggerPath "{path}"
       - For each rule (index i):
         - Build IRI: `{ns_uri}rule/{domain-slug}/{i}`
         - INSERT: rdf:type ops:Rule, ops:ruleText "{rule}", ops:priority {i}
         - Edge: domain IRI ops:hasRule rule IRI

    4. If carl_json_path provided:
       - Parse carl.json (serde_json — it's a JSON object with domains[].decisions[])
       - For each domain's decisions:
         - Build IRI: `{ns_uri}decision/{domain-slug}/{slugified-text}`
         - INSERT: rdf:type ops:Decision, ops:description "{text}", ops:rationale "{rationale}", ops:recall "{recall}"
         - Edge: domain IRI ops:hasDecision decision IRI (add ops:hasDecision to vocabulary if needed)

    5. Write back atomically (existing store pattern)
    6. Return SyncStats { domains: N, rules: N, decisions: N }

    Wire into CLI (src/cli.rs):

    - Optional `--carl <path>` flag for carl.json migration
    - Print stats on completion

    - Test: re-sync is idempotent (same triple count)
    - Test: carl.json decisions create Decision entities with domain edges

    Avoid: Modifying the existing extract/ module — this is domain-specific sync, not file extraction.
    Avoid: Complex carl.json parsing beyond domains[].decisions[]. Star commands and context brackets are later plans.
  </action>

<task type="auto">
  <name>Task 3: Graph-backed injection in user_prompt_submit</name>
  <files>src/hook/user_prompt_submit.rs, tests/graph_injection.rs</files>
  <action>
    Modify user_prompt_submit handler to query graph instead of reading DomainDef.rules:

    Current flow (replace):
    ```
    match_domains() → for each matched domain → format domain.rules → stdout
    ```

    New flow:
    ```
    match_domains() → for each matched domain →
      1. SPARQL: SELECT rules for ops:domain/{slug} via ops:hasRule
      2. SPARQL: SELECT 1-hop neighbors (decisions, projects linked via ops:hasDomain)
      3. Format: rules section (numbered, current format) + neighborhood section
      → stdout
    ```

    Implementation:
    1. After match_domains() returns matched DomainDef list:
    2. Ensure graph is loaded (call sync_domains_to_graph if not already synced this session — check mtime of domains.toml vs a sync marker file .base/.domain-sync-ts)
    3. For each matched domain:
       a. Build domain IRI from slug
       b. SPARQL SELECT: `?rule ops:ruleText ?text` WHERE `domain ops:hasRule ?rule`
       c. SPARQL SELECT: `?related ?type ?name` WHERE `{ ?related ops:hasDomain domain } UNION { domain ops:hasDecision ?related }` — get decisions, projects, entities linked to this domain
       d. Format rules in current format: `[DOMAIN_NAME] RULES:\n  0. rule1\n  1. rule2\n`
       e. Format neighborhood: `[DOMAIN_NAME] CONTEXT:\n  - Decision: {text}\n  - Project: {name} ({status})\n`
    4. Session dedup: hash the combined output (rules + neighborhood) per domain, skip if unchanged
    5. Fallback: if graph query returns empty (sync hasn't run yet), fall back to DomainDef.rules from TOML — graceful degradation, never silent failure
    6. Print combined output to stdout

    Write tests (tests/graph_injection.rs):
    - Test: matched domain's rules come from graph, not struct
    - Test: neighborhood entities appear in output
    - Test: fallback to TOML rules when graph is empty
    - Test: dedup skips re-injection of unchanged rules+context

    Avoid: Changing the matching logic itself (prompt_keywords, path, sticky). Only the injection SOURCE changes.
    Avoid: Breaking fail-open — all errors → stderr, exit 0, empty stdout.
    Avoid: Heavy SPARQL — keep to 1-2 queries per domain, not a query per rule.
  </action>
