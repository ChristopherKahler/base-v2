<objective>
## Goal
Wire AST graph data into the code navigation workflow: an explicit query command (`base ast query`) that replaces grep/glob for codebase search, automatic file-map injection on Read via PreToolUse, section-aware context on partial reads via PostToolUse, and a grep/find intercept that redirects to the graph.

## Purpose
The AST graph (289 entities, 1,074 edges from dogfood) already maps the entire codebase structure — what contains what, what calls what, what imports what. Today none of that data surfaces during code navigation. This plan makes it the default path: the graph answers "where is X?" and "what does this file contain?" before grep or blind reads ever fire.

## Output
- `base ast query` CLI subcommand (--contains, --file, --calls, --imports)
- PreToolUse on Read: automatic file entity map injection for source files
- PostToolUse on Read: section-specific entity context for partial reads
- PreToolUse on Bash: grep/find/rg intercept redirecting to `base ast query`
</objective>

<task type="auto">
  <name>Task 1: Add `base ast query` CLI subcommand</name>
  <files>src/cli.rs, src/crud/ast_query.rs (new)</files>
  <action>
    Create a new `Ast` command in the CLI with a `Query` subaction:
    
    Flags:
    - `--contains <name>` — find entities by label (case-insensitive CONTAINS match)
    - `--file <path>` — list all entities in a source file with their relationships
    - `--calls <name>` — find all callers of a named entity
    - `--imports <path>` — find all files that import from a given file
    
    Implementation in `src/crud/ast_query.rs`:
    - Load workspace graph (same pattern as other CRUD modules)
    - SPARQL queries against AST predicates: `ops:sourceFile`, `ops:sourceLine`, `ops:calls`, `ops:importsFrom`, `ops:contains`, `ops:hasMethod`, `ops:definedIn`, `rdfs:label`
    - The AST data uses `code:` prefix IRIs — queries must handle both `ops:` and `code:` namespaces
    - Output format: compact, one entity per line, relationships indented below
    
    Example output for `--contains "resolve_slug"`:
    ```
    src/crud/mod.rs:160  fn resolve_slug
      calls: build_iri, slugify, query, term_display
      called_by: cli.rs → resolve()
    ```
    
    Example output for `--file "src/crud/mod.rs"`:
    ```
    [AST] src/crud/mod.rs — 12 entities
      fn build_iri (line 21)
      fn slugify (line 48)
      fn resolve_slug (line 160)
      ...
      imports: config.rs, store.rs
      imported_by: cli.rs, project.rs, task.rs, milestone.rs
    ```
    
    Avoid: loading the full graph.trig when only ast.ttl is needed — if AST data lives in the graph.trig AST BEGIN/END block, the SPARQL query handles it (Oxigraph loads everything). Don't try to parse ast.ttl separately.
  </action>
  <verify>`base ast query --contains "slugify"` returns crud/mod.rs with line number and type</verify>
  <done>AC-1 satisfied: all four query modes return correct results from the graph</done>
</task>

<task type="auto">
  <name>Task 2: PreToolUse file map injection + grep intercept</name>
  <files>src/hook/pre_tool_use.rs</files>
  <action>
    Extend the existing PreToolUse handler with two new behaviors:
    
    **A) File map injection on Read:**
    After the existing domain-rule matching, add AST map injection:
    - Check if `tool_input.file_path` ends with a source extension (.rs, .py, .js, .ts, .go, .jsx, .tsx, .c, .cpp, .h, .java, .rb, .swift)
    - If yes, extract the relative path and query the AST graph for entities in that file
    - SPARQL: find all entities where `ops:sourceFile` matches the filename, plus `ops:importsFrom` and reverse-import edges
    - Format as compact injection block:
      ```
      [AST] {filename} — {N} entities
        Key: {top entities by type}
        Imports: {files this imports from}
        Imported by: {files that import this}
      ```
    - Session dedup: track injected files in SessionState, only inject on first Read per file per session
    
    **B) Grep/find intercept on Bash:**
    - Check if `tool_name` is "Bash" (from event JSON)
    - Check if `tool_input.command` contains grep/find/rg patterns targeting source files
    - Pattern match: `grep -r`, `grep -rn`, `find . -name`, `rg ` followed by a search term
    - If matched, inject guidance:
      ```
      <ast-hint>
      AST graph available for this workspace. Try:
        base ast query --contains "{search_term}"
      instead of grepping. The graph knows file locations, line numbers, and call relationships.
      </ast-hint>
      ```
    - Extract the search term from the grep/find command for the suggestion
    
    Important: Both behaviors are additive — existing domain rule injection continues to work. AST injection appends after domain rules.
  </action>
  <verify>
    1. Read a .rs file in the workspace — verify AST map appears in hook output
    2. Run a grep command — verify ast-hint guidance appears
  </verify>
  <done>AC-2 and AC-4 satisfied: file map auto-injects on Read, grep intercept fires on search commands</done>
</task>

<task type="auto">
  <name>Task 3: PostToolUse section-specific AST context</name>
  <files>src/hook/post_tool_use.rs</files>
  <action>
    Extend the existing PostToolUse handler:
    
    - Check if the tool was "Read" and the result contains a source file
    - Extract `file_path`, `offset`, and `limit` from the tool input (not all reads have offset/limit)
    - If offset AND limit are present (partial read):
      - Query AST graph for entities in that file where `ops:sourceLine` falls within [offset, offset+limit]
      - For each matched entity, query its `ops:calls` relationships and format:
        ```
        [AST] Lines {offset}-{offset+limit}: {entity_name} ({type})
          Calls: {called_entities with file:line}
          Called by: {callers with file:line}
        ```
    - If offset is NOT present (full read): skip (PreToolUse already injected the file map)
    - Keep existing PostToolUse behavior (lastActive timestamp update) unchanged
    
    Note: The Read tool uses `offset` as line number (1-based) and `limit` as line count. Map these to `ops:sourceLine` which is also line-number based from tree-sitter.
  </action>
  <verify>Read a file with offset=160 limit=40 — verify section-specific entity context appears in PostToolUse output</verify>
