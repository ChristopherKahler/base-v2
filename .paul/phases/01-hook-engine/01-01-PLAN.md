<objective>
## Goal
Implement the hook engine: `base hook session-start` and `base hook post-tool-use` work end-to-end — reading event JSON on stdin, loading TriG, running configurable SPARQL, emitting filtered injection to stdout. Namespace and queries are operator-configurable. All hooks fail-open.

## Purpose
This is where the binary starts doing real work. The hook engine is the runtime interface between Claude Code and the knowledge graph. Without it, the graph is inert data. With it, every session gets filtered, relevant context injected automatically — and every tool use updates the graph's activity metadata, enabling self-healing state tracking via `mtime` vs `ops:lastExtracted` comparisons in future phases.

## Output
- Config loader (`base.toml` for namespace, `queries.toml` for SPARQL queries)
- Hook dispatch module with fail-open guarantee
- Session-start handler (TriG discovery → load → query → emit)
- Post-tool-use handler (parse event → update lastActive → atomic write-back)
- Integration tests for both handlers
- Settings.json hook wiring documentation
</objective>

<acceptance_criteria>

## AC-1: Hook dispatch routes events and fails open
```gherkin
Given the binary receives `base hook session-start` or `base hook post-tool-use`
When stdin contains valid or invalid JSON (including empty stdin)
Then the correct handler is invoked for valid events,
  and on ANY error (parse failure, missing files, bad SPARQL) the process exits 0 with empty stdout
```

## AC-2: Namespace is configurable via base.toml
```gherkin
Given a `base.toml` file exists with [namespace] section specifying prefix and uri
When any hook loads the vocabulary or runs SPARQL
Then the configured namespace is used instead of the compiled default
And if no base.toml exists, the compiled default (prefix="ops", uri="http://ops-sys.local/ontology#") is used
```

## AC-3: Session-start emits filtered context from configurable queries
```gherkin
Given TriG files exist at global (~/.base-gbl/graph.trig) and/or workspace ({cwd}/.base/graph.trig) tiers
  and a queries.toml defines named SPARQL queries (or defaults are used)
When `base hook session-start` runs with cwd in stdin JSON
Then stdout contains formatted sections for each query that returns non-empty results,
  queries with empty results are silently omitted,
  and the output is structured text suitable for Claude Code system-reminder injection
```

## AC-4: Post-tool-use updates lastActive atomically
```gherkin
Given a workspace TriG file exists with project/entity data
When `base hook post-tool-use` receives event JSON with tool name and file paths on stdin
Then the relevant entities' ops:lastActive timestamps are updated via SPARQL UPDATE
  and the TriG file is atomically written (temp + rename)
  and the graph's ops:lastActive metadata enables future mtime-based extraction tracking
```

## AC-5: Queries are configurable with tiered override
```gherkin
Given embedded default queries exist in the binary
  and an operator has created ~/.base-gbl/queries.toml (global) and/or {workspace}/.base/queries.toml (workspace)
When session-start loads queries
Then workspace queries override global queries by name,
  global queries override embedded defaults by name,
  and additional queries (not in defaults) are appended in declared order
```

    - `recent-activity`: SELECT entities with lastActive in last 7 days

    - Extract `cwd` from event JSON (fall back to `std::env::current_dir()`)
    - Load `BaseConfig` from cwd
    - Route: "session-start" → `session_start::handle`, "post-tool-use" → `post_tool_use::handle`, unknown → silent exit

    **2b. Session-start handler (`src/hook/session_start.rs`):**
    Implement `pub fn handle(config: &BaseConfig, cwd: &Path) -> Result<()>`:
    1. Discover TriG files
    2. If none found → return Ok(()) (empty stdout, no error)
    3. Load all found TriG files into a single Oxigraph store via `store::load_graphs`
    4. Load vocabulary into the store (for class/predicate definitions available to queries)
    5. Load queries via `config::load_queries(cwd, config)`
    6. For each query (sorted by order):
       - Prepend `PREFIX {prefix}: <{uri}>` to the SPARQL
       - Execute via `store::query`
       - If results are non-empty, format according to query's `format` field:
         - `table`: Markdown table with column headers from SPARQL variable names
         - `list`: Bullet list, one per row
         - `prose`: Single paragraph joining values
       - Print formatted section to stdout with query description as header
    7. If ALL queries return empty results → empty stdout (silent)

    **2c. Output format:**
    Wrap all output in a clean structure:
    ```
    [section-header: {query.description}]
    {formatted results}

    ```
    Keep it minimal — Claude Code injects this as a system-reminder. No decoration, no emoji, no borders.

    **3a. Event parsing:**
    Define the expected stdin JSON shape for post-tool-use events:
    ```json
    {
      "hook": "PostToolUse",
      "tool_name": "Edit",
      "tool_input": {"file_path": "/path/to/file.rs", ...},
      "tool_output": "..."
    }
    ```
    Parse with serde into a struct. Extract file paths from `tool_input` (handle common tools: Edit, Write, Read — each has `file_path`). Gracefully handle unknown tool shapes.

    **3b. Post-tool-use handler (`src/hook/post_tool_use.rs`):**
    Implement `pub fn handle(config: &BaseConfig, cwd: &Path, event: &serde_json::Value) -> Result<()>`:
    1. Extract file paths from event JSON
    2. Discover workspace TriG file (same logic as session-start, workspace tier only — global tier is read-only for post-tool-use)
    3. If no workspace TriG exists → return Ok(()) (nothing to update)
    4. Load the workspace TriG into Oxigraph
    5. For each touched file path:
       - Resolve to a project/entity IRI if possible (check if any entity has `ops:path` matching the file's directory)
       - If match found: SPARQL UPDATE to set `ops:lastActive` to current ISO 8601 timestamp
    6. Atomic write-back via `store::write_back`
    7. stdout: empty (post-tool-use doesn't inject context, just updates the graph)

    **3c. SPARQL UPDATE for lastActive:**
    ```sparql
    DELETE { ?entity ops:lastActive ?old }
    INSERT { ?entity ops:lastActive "2026-06-01T08:42:00-05:00"^^xsd:dateTime }
    WHERE {
      ?entity ops:path ?path .
      FILTER(STRSTARTS("{file_path}", ?path))
      OPTIONAL { ?entity ops:lastActive ?old }
    }
    ```
    Use `store.update()` (Oxigraph's SPARQL UPDATE support).

    **3d. Integration test (`tests/hook_post_tool_use_test.rs`):**

    - Test with file path matching NO project → verify no error, no mutation
    - Test with missing TriG file → verify no error
