<objective>
## Goal
Implement the signal layer: parameterized SPARQL signal queries (active-awareness, pulse, staleness) with a suppression filter (dedup, novelty gate, budget cap) that replaces the Phase 1 ad-hoc query system in session-start. This is the product's core value — inject only what matters, suppress everything else.

## Purpose
Without the signal layer, session-start dumps everything in the graph. With it, the system surfaces only the relevant, novel, budget-constrained slice. This phase proves the thesis: SPARQL-filtered injection with suppression beats v1's firehose.

## Output
- Signal module with 3 signal types: active-awareness, pulse, staleness
- Suppression layer: result-hash dedup, novelty gate (only emit if data changed), budget cap (max output chars)
- Session-start integration: signals replace ad-hoc queries as the primary injection source
- Configurable signal priorities and budget via base.toml
</objective>

## AC-1: Active-awareness signal surfaces relevant entities
```gherkin
Given the graph contains projects, tasks, and decisions with lastActive timestamps
When the active-awareness signal runs
Then it returns entities active in the last 7 days, grouped by type (projects, tasks, blockers)
And results are formatted as structured sections
```

## AC-2: Pulse signal shows workspace health
```gherkin
Given the graph contains entities with various statuses and timestamps
When the pulse signal runs
Then it returns: count of active/blocked/stale items, open reminders, and recent decisions
And stale is defined as lastActive > staleThreshold days ago (default 14)
```

## AC-3: Staleness signal highlights neglected entities
```gherkin
Given entities have lastActive timestamps and staleThreshold values
When the staleness signal runs
Then it returns entities where (now - lastActive) > staleThreshold
And they are sorted by staleness (most stale first)
```

## AC-4: Suppression layer prevents redundant injection
```gherkin
Given signals were run in a previous session-start
When session-start runs again with no graph changes
Then the suppression layer detects unchanged results (hash comparison) and emits nothing
And when an entity's lastActive changes between sessions, the relevant signal IS re-emitted
```

## AC-5: Budget cap limits output size
```gherkin
Given signal config defines max_chars (default 2000)
When signals produce more output than max_chars
Then lower-priority signals are dropped and a "[truncated]" note is appended
And active-awareness (highest priority) is never dropped
```

    **1a. Signal config in base.toml (`src/config.rs`):**
    Add `SignalConfig` to BaseConfig:
    ```rust
    pub struct SignalConfig {
        pub max_chars: usize,        // budget cap (default 2000)
        pub stale_days: u32,         // staleness threshold (default 14)
        pub active_days: u32,        // active-awareness window (default 7)
        pub enabled: bool,           // master toggle (default true)
    }
    ```

    **1b. Signal module (`src/signal/mod.rs`):**
    Define the signal interface:
    ```rust
    pub struct Signal {
        pub name: String,
        pub priority: u32,       // lower = higher priority
        pub query: fn(&Path, &NamespaceConfig, &SignalConfig) -> Result<String>,
    }
    ```

    `pub fn run_signals(cwd: &Path, config: &BaseConfig) -> Result<String>`:
    1. If !config.signal.enabled → return empty
    2. Collect all signal functions, sorted by priority
    3. Run each signal → collect (name, output) pairs
    4. Apply suppression: hash results, compare with stored hashes in .base/.signal-state
    5. Apply budget cap: accumulate output until max_chars, drop lower-priority signals
    6. Return combined output string

    **1c. Suppression (`src/signal/suppression.rs`):**
    ```rust
    pub struct SignalState {
        pub hashes: HashMap<String, u64>,  // signal name → output hash
    }
    ```
    - `load(base_dir)` / `save(base_dir)` — JSON at `.base/.signal-state`
    - `is_novel(name, output_hash)` → true if hash differs from stored
    - `update(name, output_hash)` — store new hash
    - Session-start does NOT clear signal state (unlike session dedup for domains — signals persist across sessions because novelty is cross-session)

    **1d. Active-awareness signal (`src/signal/active_awareness.rs`):**
    Priority: 1 (highest — never dropped by budget)

    SPARQL query:
    ```sparql
    SELECT ?type ?name ?status ?nextAction ?lastActive WHERE {
      GRAPH ?g {
        ?entity a ?type ;
          {p}:name ?name ;
          {p}:status ?status ;
          {p}:lastActive ?lastActive .
        OPTIONAL { ?entity {p}:nextAction ?nextAction }
        FILTER(?type IN ({p}:Project, {p}:Task, {p}:Decision))
        FILTER(?lastActive > "{cutoff}"^^xsd:dateTime)
      }
    }
    ORDER BY DESC(?lastActive)
    ```

    Format output grouped by type:
    ```
    [Active Projects]
    - ProjectName (status) — next: action
    
    [Active Tasks]
    - TaskName (status, priority)
    
    [Blocked]
    - BlockedItem: reason
    ```

    **1e. Update lib.rs:** Add `pub mod signal;`
