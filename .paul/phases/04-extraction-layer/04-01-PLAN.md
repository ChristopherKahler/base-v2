<objective>
## Goal
Implement `base sync` — idempotent file-to-graph extraction. Scans workspace files, extracts metadata into RDF triples, and persists to the workspace graph. Supports two extraction plugins: markdown frontmatter (MOP) and paul.json. Uses mtime vs ops:lastExtracted for incremental-only re-extraction.

## Purpose
Without extraction, the graph only contains manually-created data (CRUD commands). Sync bridges the gap: the graph automatically reflects what files exist, their metadata, and their relationships — without the operator manually entering everything. This is the "declarative write path" from the architecture.

## Output
- `base sync` command (full scan) and `base sync --incremental` (mtime-based delta)
- Markdown frontmatter extractor (YAML frontmatter → triples)
- paul.json extractor (PAUL project metadata → triples)
- ops:lastExtracted tracking per file for incremental extraction
- Configurable include/exclude patterns via base.toml
- Integration tests verifying idempotency (re-scan = identical graph)
</objective>

## AC-1: base sync scans and extracts markdown frontmatter
```gherkin
Given a workspace contains .md files with YAML frontmatter
When base sync runs
Then each file with frontmatter is extracted as an ops:Document entity
  with ops:name, ops:path, ops:description (from frontmatter fields)
  and ops:lastExtracted set to current timestamp
And files without frontmatter are skipped silently
```

## AC-2: base sync extracts paul.json
```gherkin
Given a workspace contains .paul/paul.json files (in subdirectories)
When base sync runs
Then each paul.json is extracted as an ops:PaulProject entity
  with ops:name, ops:status, ops:path, phase info
  and linked to the workspace via ops:belongsTo
```

## AC-3: Extraction is idempotent
```gherkin
Given a file was previously extracted
When base sync runs again with no file changes
Then the graph is unchanged (same triple count, same values)
And ops:lastExtracted is NOT updated (mtime hasn't changed)
```

## AC-4: Incremental extraction uses mtime comparison
```gherkin
Given files were previously extracted with ops:lastExtracted timestamps
When base sync --incremental runs
Then only files where filesystem mtime > ops:lastExtracted are re-extracted
And unchanged files are skipped entirely
And newly created files (no ops:lastExtracted) are extracted
```

## AC-5: Include/exclude patterns are configurable
```gherkin
Given base.toml has [sync] section with include and exclude patterns
When base sync runs
Then only files matching include patterns are scanned
And files matching exclude patterns are skipped
And defaults: include = ["**/*.md", "**/paul.json"], exclude = ["node_modules/", "target/", ".git/"]
```

  <action>
    **1a. Sync config in base.toml (`src/config.rs`):**
    Add `SyncConfig` to `BaseConfig`:
    ```rust
    pub struct SyncConfig {
        pub include: Vec<String>,  // glob patterns
        pub exclude: Vec<String>,  // glob patterns
    }
    ```
    Defaults: include = `["**/*.md", "**/paul.json"]`, exclude = `["node_modules/", "target/", ".git/", ".base/"]`.
    Add `glob = "0.3"` to Cargo.toml for pattern matching.

    **1b. Extraction module (`src/extract/mod.rs`):**
    ```rust
    pub mod frontmatter;
    pub mod paul_json;
    ```

    Core sync function:
    `pub fn sync(cwd: &Path, ns: &NamespaceConfig, sync_config: &SyncConfig, incremental: bool) -> Result<SyncReport>`

    Algorithm:
    1. Walk workspace directory (skip excluded patterns)
    2. For each file matching include patterns:
       a. Compute file IRI from path: `{ns.uri}document/{path-hash}` (hash of relative path)
       b. If incremental: check mtime vs ops:lastExtracted — skip if unchanged
       c. Route to extractor by file type (.md → frontmatter, paul.json → paul_json)
       d. Extractor returns a Vec of (predicate, value) triples
       e. DELETE existing triples for this file IRI, INSERT new triples
       f. Set ops:lastExtracted to current timestamp
    3. Return SyncReport { scanned: usize, extracted: usize, skipped: usize }

    **File IRI scheme:** Use the relative path as the slug (with `/` → `-` and `.` → `-`):
    `{ns.uri}document/{relative-path-slug}`
    Example: `planning/base-v2/PLANNING.md` → `{ns.uri}document/planning-base-v2-planning-md`

    **1c. Frontmatter extractor (`src/extract/frontmatter.rs`):**
    `pub fn extract(content: &str, file_path: &str) -> Option<Vec<(String, String)>>`

    Parse YAML frontmatter between `---` delimiters. Extract key-value pairs as triples:
    - `name` or `title` → ops:name
    - `description` → ops:description
    - `status` → ops:status
    - `tags` (array) → multiple ops:description triples (or a joined string)
    - All other string fields → ops:description with field name prefix

    Always include: ops:path = file_path, rdf:type = ops:Document.

    Use basic string parsing for YAML (split on `---`, parse `key: value` lines). No heavy YAML dependency — frontmatter is simple key-value.

    **1d. Add `pub mod extract;` to lib.rs.**

    **1e. Add `file_iri_from_path` helper to crud/mod.rs** for consistent path→IRI conversion.

<task type="auto">
  <name>Task 2: paul.json extractor + incremental logic</name>
  <files>src/extract/paul_json.rs, src/extract/mod.rs</files>
  <action>
    **2a. paul.json extractor (`src/extract/paul_json.rs`):**
    `pub fn extract(content: &str, file_path: &str) -> Option<Vec<(String, String)>>`

    Parse paul.json (already valid JSON via serde_json). Map fields:
    - `name` → ops:name
    - `version` → ops:description (version string)
    - `phase.name` → ops:status (current phase name)
    - `phase.status` → ops:status (phase status)
    - `milestone.name` → ops:description (milestone)
    - `timestamps.updated_at` → ops:lastActive
    - File path → ops:path
    - Type → ops:PaulProject

    **2b. Incremental logic in sync function:**
    Before extracting, check if the file was previously extracted:
    ```sparql
    SELECT ?lastExtracted WHERE {
      GRAPH ?g { <file_iri> ops:lastExtracted ?lastExtracted }
    }
    ```
    Compare with file mtime (as ISO 8601). If mtime <= lastExtracted, skip.

    For mtime, use `std::fs::metadata(path)?.modified()?` → convert to chrono DateTime → compare.

    **2c. DELETE before INSERT for re-extraction:**
    When re-extracting a file, first delete ALL existing triples for that file IRI:
    ```sparql
    DELETE WHERE { GRAPH ?g { <file_iri> ?p ?o } }
    ```
    Then INSERT the fresh triples. This ensures idempotency — re-extraction produces identical results.
