<objective>
## Goal
Implement deterministic domain matching via `domains.toml` with multi-signal matching (keywords + file paths + always-on + sticky), wire `base hook user-prompt-submit` to inject matched rules to stdout, session-level dedup to prevent re-injection, and `base domain add-trigger` for CLI domain mutation.

## Purpose
This replaces CARL's Python domain matching with a Rust implementation. The domain matcher is the first piece of CARL absorption — it determines WHAT context gets injected based on what the operator is doing. Combined with Phase 1's hook engine, this completes the read-side context injection pipeline.

## Output
- Domain config parser (`domains.toml` with tiered loading)
- Multi-signal matcher (keywords, file paths, always-on, sticky)
- `base hook user-prompt-submit` handler
- Session-level dedup via `.base/.session` state
- `base domain add-trigger` CLI command
- Integration tests
</objective>

## AC-1: domains.toml parsed with tiered loading
```gherkin
Given a domains.toml exists at workspace (.base/domains.toml) and/or global (~/.base-gbl/domains.toml)
When the domain config is loaded
Then domains merge (workspace overlays global by name), each domain has name, mode, keywords, paths, exclude, sticky, and rules fields
And if no domains.toml exists anywhere, an empty domain set is returned (no error)
```

## AC-2: Multi-signal matcher identifies domains from prompt and context
```gherkin
Given domains.toml defines domains with keyword and path triggers
When a user prompt contains a keyword matching a domain trigger
Then that domain is matched
And when a recently-active file path (from graph lastActive) matches a domain path trigger
Then that domain is also matched
And domains with mode="always" are always matched regardless of signals
And domains with sticky=true remain matched for the rest of the session once triggered
```

## AC-3: user-prompt-submit emits matched rules to stdout
```gherkin
Given matched domains contain rules
When base hook user-prompt-submit runs
Then stdout contains the matched domains' rules formatted for Claude Code injection
And the output groups rules by domain with domain name as header
And if no domains match, stdout is empty (silent)
```

## AC-4: Dedup prevents re-injection within a session
```gherkin
Given a domain was injected earlier in the current session
When the same domain matches again on a subsequent prompt
Then its rules are NOT re-injected (dedup by domain name)
And if the domain's rules have CHANGED since last injection (file modified), they ARE re-injected
And session-start clears the dedup state (fresh session = fresh injection)
```

## AC-5: base domain add-trigger mutates domains.toml
```gherkin
Given a workspace domains.toml exists (or is created)
When the operator runs base domain add-trigger --domain <name> --keyword <kw>
Then the keyword is appended to the domain's keywords array
And if the domain doesn't exist, it is created with mode="triggered" and the keyword
And the file is written atomically
```

    ```rust
    pub struct DomainDef {
        pub name: String,
        pub mode: DomainMode,          // Always | Triggered
        pub keywords: Vec<String>,      // keyword triggers (case-insensitive match)
        pub paths: Vec<String>,         // path glob triggers
        pub exclude: Vec<String>,       // exclusion patterns
        pub sticky: bool,              // stays active once triggered
        pub rules: Vec<String>,         // rule text
    }
    pub enum DomainMode { Always, Triggered }
    ```

    Implement `load_domains(cwd: &Path) -> Vec<DomainDef>`:
    - Load global `~/.base-gbl/domains.toml` (if exists)
    - Layer workspace `.base/domains.toml` on top (override by name, append new)
    - If neither exists, return empty Vec (no error)
    - Same merge-by-name pattern as queries.toml

    The TOML format:
    ```toml
    [[domain]]
    name = "global"
    mode = "always"
    rules = [
      "Rule text 1",
      "Rule text 2",
    ]

    [[domain]]
    name = "development"
    mode = "triggered"
    sticky = true
    keywords = ["write code", "fix bug", "implement", "refactor"]
    paths = ["src/", "lib/", "tests/"]
    rules = [
      "Follow existing code patterns.",
    ]
    ```

    Create a minimal session state tracker:

    ```rust
    pub struct SessionState {
        pub injected_domains: HashMap<String, u64>,  // domain name → rules hash
    }
    ```

    File: `.base/.session` (JSON, single object)
    - `SessionState::load(cwd: &Path) -> SessionState` — reads file, returns empty state if missing
    - `SessionState::save(&self, cwd: &Path) -> Result<()>` — atomic write
    - `SessionState::clear(cwd: &Path) -> Result<()>` — truncate (called by session-start)
    - `SessionState::is_injected(&self, domain: &str, rules_hash: u64) -> bool` — true if domain injected with same hash
    - `SessionState::mark_injected(&mut self, domain: &str, rules_hash: u64)` — record injection

    The rules_hash is a simple hash of the concatenated rule texts. If rules change (domains.toml edited), hash differs → re-inject.

  <name>Task 2: Multi-signal matcher + user-prompt-submit handler</name>
  <files>src/domain/matcher.rs, src/hook/user_prompt_submit.rs, src/hook/mod.rs, tests/domain_matcher_test.rs, tests/hook_user_prompt_submit_test.rs</files>
  <action>
    **2a. Matcher (`src/domain/matcher.rs`):**
    Implement `match_domains(prompt: &str, domains: &[DomainDef], session: &SessionState, active_paths: &[String]) -> Vec<&DomainDef>`:

    Algorithm:
    1. Collect all `mode=Always` domains → always matched
    2. For each `mode=Triggered` domain:
       a. **Keyword match:** check if any keyword appears in prompt (case-insensitive substring)
       b. **Path match:** check if any path in `active_paths` starts with any domain path trigger
       c. **Sticky:** check if domain name is in session state (already triggered this session)
       d. **Exclude:** if any exclude pattern matches prompt or paths, skip domain
       e. If any signal (a, b, c) fires and no exclude matches → domain matched
    3. For matched domains, check dedup: `session.is_injected(domain, hash)` — skip if already injected with same hash
    4. Return list of domains to inject (new or changed since last injection)

    **2b. user-prompt-submit handler (`src/hook/user_prompt_submit.rs`):**
    Implement `pub fn handle(config: &BaseConfig, cwd: &Path, event: &serde_json::Value) -> Result<()>`:
    1. Extract prompt text from event JSON (`event.tool_input.prompt` or `event.prompt`)
    2. Load domains via `domain::load_domains(cwd)`
    3. Load session state via `SessionState::load(cwd)`
    4. Optionally load graph for active file paths:
       - Find workspace TriG, load graph
       - Query for entities with recent `ops:lastActive` (last 1 hour)
       - Extract their `ops:path` values as `active_paths`
       - If no TriG or query fails: empty active_paths (graceful)
    5. Run matcher: `match_domains(prompt, &domains, &session, &active_paths)`
    6. Format output:
       ```
       [DOMAIN: {name}]
       {rule 1}
       {rule 2}

       [DOMAIN: {name}]
       {rule 1}
       ```
       Minimal format — no decoration, no emoji. Domain name as section header, rules as plain text.
    7. Update session state: mark all newly injected domains
    8. Save session state
    9. Print formatted output to stdout

    **2c. Wire handler in hook/mod.rs:**
    Replace user-prompt-submit stub with actual `user_prompt_submit::handle` call.

    **2d. Integration tests:**

    `tests/domain_matcher_test.rs`:
    - Keyword match: prompt contains trigger → domain matched
    - No match: prompt has no triggers → empty result
    - Always-on: mode=always matched regardless
    - Sticky: domain matched once → stays matched on subsequent calls
    - Exclude: domain would match but exclude pattern fires → not matched
    - Dedup: domain already injected with same hash → not in results
    - Dedup override: domain rules changed (different hash) → re-injected
    - Path match: active_paths matches domain path trigger → matched

    `tests/hook_user_prompt_submit_test.rs`:
    - Full handler test: create domains.toml + prompt → verify stdout contains rules
    - No domains.toml → silent (empty stdout)
    - Fail-open: malformed domains.toml → no crash
