<objective>
## Goal
Implement CRUD infrastructure (SPARQL mutation helpers, IRI generation, supersession) and the first two entity types — projects and tasks — as CLI commands. Establishes the pattern that Plan 03-02 replicates for remaining entity types.

## Purpose
Without CRUD commands, the graph has no operator-managed data. This phase bridges the gap: operators can create projects, track tasks, and the graph reflects their actual working state. Session-start queries (Phase 1) and domain matching (Phase 2) then have real data to surface.

## Output
- CRUD infrastructure: SPARQL mutation helpers, IRI builder, graph write pipeline
- `base project add/list/get/update` — full project lifecycle
- `base task add/done/list` — task management within projects
- Integration tests for both entity types
</objective>

## AC-1: SPARQL mutation helpers work correctly
```gherkin
Given the CRUD infrastructure module exists
When a caller builds an INSERT or DELETE+INSERT mutation
Then the SPARQL is valid, executes against Oxigraph, and the triple store reflects the change
And write_back atomically persists the updated graph to TriG
```

## AC-2: IRI generation follows the ontology scheme
```gherkin
Given a namespace config and an entity type + slug
When an IRI is generated
Then it follows the pattern {namespace_uri}{type}/{slug} (e.g., http://ops-sys.local/ontology#project/casegate-v2)
And the slug is derived from the name (lowercased, spaces→hyphens, special chars stripped)
```

## AC-3: Project CRUD works end-to-end
```gherkin
Given the workspace has a .base/ directory
When the operator runs base project add --name "My Project" --status active
Then a new project entity is created in the workspace graph with ops:Project type, name, status, createdAt, path
And base project list shows the project in a table
And base project get my-project shows full details
And base project update my-project --status blocked --blocked-by "reason" updates the status
```

## AC-4: Task CRUD works end-to-end
```gherkin
Given a project exists in the workspace graph
When the operator runs base task add --project my-project --name "Fix auth" --priority high
Then a new task is created with ops:Task type, linked to the project via ops:hasTask
And base task list --project my-project shows tasks for that project
And base task done fix-auth sets status to completed with a completedAt timestamp
```

## AC-5: Supersession preserves audit trail for status changes
```gherkin
Given a project has status "active"
When the operator updates it to status "blocked"
Then the old status triple is replaced with the new status
And an ops:updatedAt timestamp is set
And the project's ops:lastActive is refreshed
```

<task type="auto">
  <name>Task 1: CRUD infrastructure module</name>
  <files>src/crud/mod.rs, src/lib.rs</files>
  <action>
    **1a. Create CRUD module (`src/crud/mod.rs`):**

    Shared helpers for all entity CRUD operations:

    ```rust
    pub mod project;
    pub mod task;
    ```

    **IRI builder:**
    `pub fn build_iri(ns: &NamespaceConfig, entity_type: &str, slug: &str) -> String`
    - Returns `{ns.uri}{entity_type}/{slug}`
    - Example: `http://ops-sys.local/ontology#project/my-project`

    **Slug generator:**
    `pub fn slugify(name: &str) -> String`
    - Lowercase, spaces → hyphens, strip non-alphanumeric (except hyphens)
    - "My Cool Project" → "my-cool-project"

    **SPARQL INSERT helper:**
    `pub fn sparql_insert(graph_iri: &str, subject: &str, triples: &[(String, String)], ns: &NamespaceConfig) -> String`
    - Builds `INSERT DATA { GRAPH <{graph_iri}> { <{subject}> {pred} {obj} . } }`
    - Each triple is (predicate_local_name, value) — auto-wraps strings in quotes, handles typed literals

    **SPARQL DELETE+INSERT helper for updates:**
    `pub fn sparql_update_field(graph_iri: &str, subject: &str, pred: &str, new_value: &str, ns: &NamespaceConfig) -> String`
    - DELETE old value, INSERT new value for a specific predicate

    **Graph write pipeline:**
    `pub fn load_and_mutate(cwd: &Path, ns: &NamespaceConfig, sparql: &str) -> Result<()>`
    - Find workspace .base/graph.trig (create if missing with empty named graph)
    - Load into Oxigraph
    - Execute SPARQL UPDATE
    - Atomic write_back

    **Graph query pipeline:**
    `pub fn load_and_query(cwd: &Path, sparql: &str) -> Result<QueryResults>`
    - Find workspace TriG, load, query, return results

    **Workspace graph IRI:**
    `pub fn workspace_graph_iri(ns: &NamespaceConfig, workspace_slug: &str) -> String`
    - Returns `{ns.uri}graph/ws/{workspace_slug}`
    - Workspace slug derived from the workspace directory name

    **Timestamp helper:**
    `pub fn now_iso() -> String` — current time as ISO 8601 with timezone

    **1b. Update lib.rs:**
    Add `pub mod crud;`.
  </action>

    `cargo test` passes (existing tests green).
  </verify>

<task type="auto">
  <name>Task 2: Project CRUD commands</name>
  <files>src/crud/project.rs, src/cli.rs, tests/crud_project_test.rs</files>
  <action>
    **2a. Project module (`src/crud/project.rs`):**

    `pub fn add(cwd: &Path, ns: &NamespaceConfig, name: &str, status: &str, path: Option<&str>) -> Result<String>`
    - Generate slug from name
    - Build IRI: `{ns.uri}project/{slug}`
    - Determine workspace graph IRI from cwd directory name
    - Build SPARQL INSERT with triples:
      - `rdf:type ops:Project`
      - `ops:name "{name}"`
      - `ops:status "{status}"`
      - `ops:createdAt "{now}"^^xsd:dateTime`
      - `ops:lastActive "{now}"^^xsd:dateTime`
      - `ops:path "{path}"` (if provided, default to cwd)
      - `ops:belongsTo <workspace_iri>`
    - Execute via load_and_mutate
    - Return slug

    `pub fn list(cwd: &Path, ns: &NamespaceConfig) -> Result<()>`
    - SPARQL SELECT: name, status, priority, lastActive from all ops:Project entities
    - Format as table to stdout

    `pub fn get(cwd: &Path, ns: &NamespaceConfig, slug: &str) -> Result<()>`
    - SPARQL SELECT: all properties of the specific project IRI
    - Format as key-value pairs to stdout

    `pub fn update(cwd: &Path, ns: &NamespaceConfig, slug: &str, status: Option<&str>, blocked_by: Option<&str>, next_action: Option<&str>) -> Result<()>`
    - For each provided field: build DELETE+INSERT SPARQL
    - Always update ops:updatedAt and ops:lastActive
    - Execute all updates in a single load → mutate → write_back cycle

    **2b. Wire CLI (`src/cli.rs`):**
    Replace Project stub with real ProjectAction subcommand:
    ```rust

        Add { name: String, status: Option<String>, path: Option<String> },

        Get { slug: String },
        Update { slug: String, status: Option<String>, blocked_by: Option<String>, next_action: Option<String> },
    }
    ```

    **2c. Integration test (`tests/crud_project_test.rs`):**
    - Add project → verify TriG contains project triples
    - List projects → verify output
    - Get project by slug → verify detail output
    - Update project status → verify new status in TriG, old status gone
    - Add project with path → verify ops:path set
  </action>

    **3a. Task module (`src/crud/task.rs`):**

    `pub fn add(cwd: &Path, ns: &NamespaceConfig, project_slug: &str, name: &str, priority: Option<&str>) -> Result<String>`
    - Generate slug: `{project_slug}.{task_slug}` (dot-separated per IRI scheme in ops.ttl)
    - Build IRI: `{ns.uri}task/{project_slug}.{task_slug}`
    - SPARQL INSERT: type, name, status="active", priority, createdAt, lastActive
    - Also INSERT `<project_iri> ops:hasTask <task_iri>` to link task to project
    - Execute via load_and_mutate

    `pub fn list(cwd: &Path, ns: &NamespaceConfig, project_slug: Option<&str>) -> Result<()>`
    - If project_slug provided: SELECT tasks WHERE project ops:hasTask ?task
    - If not provided: SELECT all tasks
    - Format as table

    `pub fn done(cwd: &Path, ns: &NamespaceConfig, slug: &str) -> Result<()>`
    - DELETE+INSERT status from current → "completed"
    - INSERT ops:updatedAt with current timestamp
    - Update ops:lastActive
