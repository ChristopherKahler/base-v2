# BASE v2 Codebase Audit Report

**Codebase:** ~14K lines of Rust | **Modules reviewed:** 8 | **Date:** 2026-06-09

---

## 1. EXECUTIVE SUMMARY

BASE v2 is a well-architected RDF knowledge graph engine with Claude Code hook integration, but it has 6 data-corruption-class bugs in its SPARQL generation layer (missing escaping, property mismatches, index collisions) and a crash bug in the hook system from unsafe UTF-8 slicing. The project has shipped 16 phases in 10 days with strong velocity, but process tracking has degraded since Phase 14 -- Phase 16 code was committed without any PAUL plan/summary artifacts, and PROJECT.md is 8 days stale. The highest-impact fix is consolidating SPARQL escaping into a single function and applying it to the 6 modules that skip it, which would close the entire class of data corruption bugs in one pass.

## 3. CRITICAL ISSUES (broken or will break)

### C1. CRASH BUG: UTF-8 byte slice panic
**File:** `src/hook/user_prompt_submit.rs:229`
```rust
Some(format!("{}...", &prompt[..117]))
```
Slices by byte index without char boundary check. Any prompt with multi-byte characters (emoji, CJK, accented) near byte 117 panics with `byte index is not a char boundary`. This crashes the user-prompt-submit hook, which means Claude Code receives no context injection for that prompt.
**Fix:** Replace with `prompt.chars().take(117).collect::<String>()` or `prompt.char_indices().take_while(|(i,_)| *i < 117)`.

### C2. DATA MISMATCH: Decision recall is silently broken
**Files:** `src/crud/decision.rs:41` writes decision text as `{p}:name`. `src/crud/note.rs:101` queries decisions via `{p}:description`. The UNION branch that searches decision text will never return results because the predicate doesn't match. The rationale branch (line 109, `{p}:rationale`) does work, so partial recall functions -- but searching for a decision by its actual decision text through `base recall` returns nothing.
**Fix:** Change `note.rs:101` from `{p}:description` to `{p}:name`, or add a third UNION branch for `{p}:name`.

### C3. INDEX COLLISION: Rule add after delete
**File:** `src/crud/rule.rs:126-145`
`next_rule_index()` uses `COUNT` of existing rules to determine the next index. After `remove()` deletes a rule, COUNT decreases, causing the next `add()` to reuse an index that collides with a still-existing rule IRI. Example: rules [0,1,2] exist; remove rule 1; COUNT=2; next add creates rule at index 2, overwriting existing rule 2.
**Fix:** Use `MAX(?idx) + 1` instead of `COUNT`, or parse existing indices and find the first gap/next-after-max.

### C4. SPARQL INJECTION: Dashboard search endpoint
**File:** `src/dashboard/api.rs:218-234`
User-supplied query string is interpolated directly into SPARQL via `format!()` after lowercasing but without escaping quotes. A search containing `"` breaks the SPARQL or injects arbitrary patterns. While Oxigraph is in-process (not remote), malformed queries cause silent failures or unexpected behavior.
**Fix:** Apply `escape_sparql_literal()` to `q_lower` before interpolation.

### C5. MISSING SPARQL ESCAPING: goal.rs and reminder.rs
**Files:** `src/crud/goal.rs:21-22`, `src/crud/reminder.rs:21-22`
Both `add()` functions interpolate `name`, `target`, and `due_date` directly into SPARQL string literals without any escaping. A goal name containing `"` or `\` produces malformed SPARQL, causing silent data loss (the `load_and_mutate` error propagates but the CLI's `let _ =` pattern in several callers would swallow it).
**Fix:** Apply `escape_sparql_literal()` to all user-supplied string values before interpolation.

### C6. MISSING SPARQL ESCAPING: ledger.rs
**File:** `src/extract/ledger.rs:75-87`
Fields `action` (line 76), `phase` (line 84), and `plan` (line 87) are interpolated into SPARQL without escaping. The `note` field (line 93) gets partial escaping (only `"` and `\n`). A ledger entry with a double-quote in the action field produces broken SPARQL that silently fails (guarded by `is_ok()` at line 115).
**Fix:** Use the `escape()` function already defined in `paul_toml.rs:242` (or the centralized version once extracted).

### C7. SILENT ERROR SWALLOWING: 16 discarded Results in CLI
**File:** `src/cli.rs:471-857` (16 occurrences)
Pattern `let _ = crud::*()` discards `Result` values from list/get/search/recall operations. If these functions return `Err`, the error is swallowed and the CLI exits 0, giving the user no indication anything went wrong. Affected: ast query (4), project list/get, milestone list/get, task list, decision search, entity list/get, goal list, reminder list, recall.
**Fix:** Replace `let _ =` with `if let Err(e) = ... { eprintln!("Error: {e}"); }` consistently.

### C8. FAILING TEST: paul_json.rs
**File:** `src/extract/paul_json.rs:69`
Test asserts triples contain `"PaulProject"` but the extractor emits `"{p}:Project"` (line 14). With default namespace prefix `ops`, the emitted value is `ops:Project`. The test has been broken since the type was unified with paul_toml. This is a `cargo test` failure.
**Fix:** Change assertion from `"PaulProject"` to `"Project"`.

### C9. DOUBLE DEDUP with mismatched hashes
**Files:** `src/domain/matcher.rs:50-54` and `src/hook/user_prompt_submit.rs:154`
Dedup is checked twice per domain with different hash inputs. The matcher hashes `rules + query_name`. The hook re-hashes the fully rendered output (rules + neighborhood + SPARQL results). These hashes almost never match, making the matcher's dedup check ineffective when graph-backed injection adds neighborhood context. The matcher passes domains through that the hook then re-deduplicates, wasting graph queries and rendering work.
**Fix:** Remove the matcher-level dedup (let the hook's more accurate hash be the single dedup gate), or unify the hash input so both levels use the same data.

### C10. GRAPH CORRUPTION: `sync --ast` appends raw Turtle into graph.nq *(found 2026-06-10, post-audit)*
**Files:** `src/cli.rs:710-735`
`base sync --ast` appends the Python extractor's Turtle output (`ast.ttl`) directly into `graph.nq` between `# --- AST BEGIN/END ---` comment markers. In the TriG era this parsed (TriG is a Turtle superset); since the NQuads switch (`bebb20c`) it corrupts the entire graph file — every subsequent `load_graph` fails, all hooks go silent (fail-open), and every CRUD command errors. Hit live in chris-ai-systems workspace 2026-06-09→10; repaired by stripping the block (backup: `.base/graph.nq.bak-2026-06-10`).
**Fix:** Stop appending Turtle to graph.nq. Either (a) keep AST in `ast.ttl` as a sibling and load it as an additional graph in `load_merged_graph`/`load_graphs` with `RdfFormat::Turtle`, or (b) parse the TTL and re-serialize as NQuads quads under a dedicated named graph before merging. (a) is cleaner — AST regenerates wholesale, doesn't belong in the CRUD-managed file.
**Related:** C11 — `migrate_trig_to_nq` deleted its own input when passed a `.trig` path (`trig_path == nq_path`); fixed during Phase 17 UNIFY (`f31d10b`).

## 6. QUALITY GAPS

### Q1. Inconsistent SPARQL escaping (systemic)
Three independent escape implementations exist: `crud/mod.rs:escape_sparql_literal()` (centralized), manual inline in `note.rs:26` and `rule.rs:26`, and identical `escape()` functions duplicated across `frontmatter.rs:369`, `paul_md.rs:455`, `paul_toml.rs:242`. Six modules skip escaping entirely (goal, reminder, ledger action/phase/plan, decision search keyword, note recall keyword, ast_query search terms). This is the single largest systemic quality gap.

### Q2. Graph loaded from disk multiple times per hook invocation
`user_prompt_submit` loads the graph 2x (lines 78, 668). `pre_tool_use` loads it up to 3x (lines 64, 159, plus indirect). Each load parses the full NQuads file into a fresh Oxigraph store that is discarded after the hook exits. Should consolidate to a single load per invocation.

### Q3. Multiple independent SessionState instances in pre_tool_use
**File:** `src/hook/pre_tool_use.rs:51` and `src/hook/pre_tool_use.rs:139`
Two SessionState objects load/save independently. Mutations to one don't affect the other. Creates unnecessary disk I/O and could cause lost writes if both branches save with different mutations.

### Q4. Exclude patterns don't apply to file-path matches
**File:** `src/domain/matcher.rs:72-77`
Exclude patterns check only against `prompt_lower`, not against file paths. A domain triggered by path matching cannot be vetoed by exclude patterns based on path content. Additionally, always-on domains (line 68-69) bypass exclude entirely.

### Q5. Substring keyword matching with no word boundaries
**File:** `src/domain/matcher.rs:80-83`
`prompt_lower.contains(&kw.to_lowercase())` -- a keyword like `"go"` matches `"going"`, `"google"`, `"algorithm"`. Short keywords produce frequent false-positive domain matches.

### Q6. Unsafe cwd fallback
**File:** `src/cli.rs:461`
`std::env::current_dir().unwrap_or_default()` silently falls back to an empty PathBuf. Every subsequent file operation would fail silently or operate on undefined relative paths.

### Q7. Silent error swallowing in signal system
**File:** `src/signal/mod.rs:46,53,73`
All signal `Err(_)` results are discarded with `Err(_) => {}`. No logging, no diagnostics. Contradicts the codebase's own stated principle against silent error suppression.

### Q8. post_tool_use rewrites entire graph.nq on every file touch
**File:** `src/hook/post_tool_use.rs:42`
`lastActive` timestamp updates run SPARQL UPDATE then `store::write_back()`, serializing the entire graph to disk on every post-tool-use event that touches a file with a matching entity.

### Q9. Duplicate staleness logic
**Files:** `src/signal/staleness.rs` (P3) and `src/signal/flow_resurface.rs:104-160` (P2)
Both query for active entities past a staleness threshold, both defaulting to 14 days. When flow is enabled, the same entities appear in both output sections.

### Q10. Install step numbering bug
**File:** `src/install.rs:42-46, 505, 687`
Two steps both labeled "Step 6" in comments. User-facing output prints steps 1,2,3,4,5,5,7 -- skipping 6.

### Q11. Fragile workspace registration
**File:** `src/scaffold.rs:130`
Uses `content.contains(path)` for dedup. Path `/home/user/project` falsely matches `/home/user/project-v2`.
